# Unity - From Zero

## Контроль версий

- Настроить сериализацию Unity, `.gitignore`, при необходимости создать `.gitattributes`
- Ни один коммит не должен содержать артефакты или файлы из несвязанной ветки
- Коммиты должны быть разделены по контексту

## Управление и навигация

- Использовать [Input System]. При необходимости создавать несколько `Action maps` (Player/UI) и переключать их в зависимости от состояния игры
- Использовать навигацию Unity (на сцене должна быть одна активная `EventSystem`)

## Графика и оптимизация

- Использовать только ассеты с поддержкой _[HDRP]_ и _URP_. Сохранять исходные материалы, декали и шейдеры для быстрого переключения рендер-пайплайна
- Разделять коллизию и LOD в окружении. Использовать [LOD] только при необходимости. Коллизии по возможности строить из примитивов и разделять физику по слоям
- По возможности не использовать _Odin Inspector_ и Dependency Injection через _Zenject_ (чёрный экран дольше 40 секунд на Nintendo Switch при запуске в `Slower Mode`)

## Пользовательский интерфейс

- Использовать композитные префабы везде, где это возможно (включая UI). Не изменять параметры префабов на сцене. Создавать модульные префабы со связями через `ScriptableObjects` или абстракции
- Разделять префабы на части, если в них более 8 дочерних объектов. По возможности использовать стандартные компоненты Unity вместо кастомных решений
- Элементы UX/UI что заменятся при смене платформы должны быть динамическими. Настроить TMP-ассеты под каждый шрифт и вес. Сохранить ссылку на Figma-дизайн

## Вызов сторонних API

- Ни одна сцена в билде не должна содержать консольных ошибок. Ограничивать платформенно-специфичные вызовы через препроцесс-директивы (`#if UNITY_SWITCH`, `#if UNITY_PS4`, `#if UNITY_PS5`), чтобы исключить ошибки компиляции при смене платформы
- Новый функционал следует проверять в Unity Editor
- Удалять лишние логи. Оставлять только минимальный [`Debug.Log`][Logging], отражающий последовательность ключевых вызовов

## Менеджмент проектов

- В папке готового проекта не должно быть лишних файлов. В коде не должно быть лишних комментариев или [неиспользуемых методов][YAGNI]
- Вся _релевантная информация_ по проекту (дизайн-идея, известные нерешённые баги) должна передаваться PM вместе со ссылкой на GitHub-репозиторий и ветку
